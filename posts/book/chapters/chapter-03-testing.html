Testing

The need of testing

First we will argue that testing is a way to evolve without the pain of evolution To understand the need for testing, first we need to understand that path to creation is never a straight line path.

Mistakes are a part of creation process

Creators iterate endlessly, learning from their previous mistakes all the time, and repeatedly improving their product. A great product works so flawlessly and obviously that it is easy to think that it is as trivial to create it as it is to use it.

Air travel is very reliable these days, but countless people have lost their lives, including aeroplane pioneers who dared to fly on their budding aeroplanes. We have learnt from automobile accidents, collapse of bridges and so on.

Science books tell us about the laws of the nature, but do not describe how erratic the process of deriving them was.

Nature (or God for the believers) itself creates through trial and error. The bodies of human beings and other animals are products of billions of years of evolution. Nature never ceases to experiment, and some of the genetic mutations lead to diseases, and the lives of affected individuals are destroyed. But that is how nature works.

Making errors is inevitable. Some people implicitly believe that making error is a sign of weakness, something which is not desirable. While they may be true, we cannot do anything about it. Thus, we need to build systems which are take into account that we will make mistakes.

Make mistakes while not suffering conseuquences

Testing allow us to develop software so that we can make mistakes without impacting real world. This speeds up development process: we can write code without being fully confident that we are writing correct code, since testing allows us to find errors and gives us a chance to correct the code, before we ship it to our customers.

The testing mindset

To be able to build a robust software, you will need to test it. In order for you to be able to test it, you need to have a testing mindset.

The testing mindset consists of the following: I accept that the code I write is likely to be incorrect. Hence I will write code in such a way that I can test it, and then I will test it thoroughly.

In practice, the testing mindset implies the following

    We write testable code: in a way that later we can test is easily.
    We write test cases to test key functionalities to automate testing.
    We perform integration testing before we go live with our product.
    We perform testing on the live product after we have gone live.

Writing testable code

Writing code is an art. The more you get better at it, the more you are able to appreicate and write beautiful code.

One of the characterstics of a well written code is that it is testable. It is easily possible to write test cases. Note what the the testing mindset describes above says that you need to write code such that you can test it later. It implies that testing cannot be an add-on once you have finished development. Rather, testing needs to be "baked in" during the development process itself.

When beginning to write the code, you must ask yourself: "How will I be able to test this code". The code should be written in a way that it lends itself to testing.
